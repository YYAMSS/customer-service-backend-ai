# 电商业务服务（ecommerce-service-backend）技术点与架构实现要点

> 目的：总结 `ecommerce-service-backend/` 的架构分层、技术点、实现方式与编码风格，供客服系统联调与同类服务复用参考。

## 1. 技术栈与核心依赖

- **Web 框架**：FastAPI（同步路由函数为主，OpenAPI Tags 分组）。
- **ASGI Server**：Uvicorn（开发模式 `reload=True`）。
- **ORM**：SQLAlchemy 2.x（同步 `Engine` + `Session`，`relationship` + `joinedload` 做聚合查询）。
- **MySQL 驱动**：PyMySQL（`mysql+pymysql://...`）。
- **安全依赖**：`cryptography`（为 MySQL/SSL 等底层依赖提供支持）。
- **依赖管理**：`pyproject.toml` + `uv.lock`（`[tool.uv] package=false`，偏“应用工程”而非可发布包）。

## 2. 工程形态与入口

- **工程目录**：`ecommerce-service-backend/`
- **启动入口**：
  - `main.py`：打印启动信息后，`uvicorn.run("app.app:app", ...)`。
  - `app/app.py`：构建 FastAPI `app`，定义 OpenAPI tags，挂载 `app/api.py` 的路由。
- **环境变量**：`.env.example`（README 指向）+ `app/config.py` 中通过 `os.getenv` 读取：
  - `DATABASE_URL`（默认连本机 `commerce` 库）
  - `APP_HOST`（默认 `0.0.0.0`）
  - `APP_PORT`（默认 `18081`）

## 3. 分层结构（从外到内）

### 3.1 `app/app.py`：应用装配（App Composition）

- 定义 `openapi_tags`（系统/用户/订单/商品）以提升 docs 可读性。
- `app.include_router(router)` 将业务路由集中挂载到根应用。

### 3.2 `app/api.py`：API 路由 + 业务查询（单文件聚合）

该服务将“路由 + 查询 + DTO 组装”集中在 `app/api.py`，常见模式：

- **依赖注入 DB Session**：每个接口 `db: Session = Depends(get_db)`
- **统一响应包装**：`ApiResponse(code=0,message="ok",data=...)`，通过 `_wrap(data)` 返回
- **404/业务错误**：使用 `HTTPException`（如用户不存在/订单不存在/状态不允许/退款冲突等）
- **聚合查询**：
  - 用户最近订单：`Order` + `OrderItem` + `Product` 的 `joinedload`，按创建时间倒序取 `limit`
  - 用户最近商品：从 `OrderItem` join 到 `Order`/`Product`，再去重 `product_id` 保持列表精炼
- **订单操作**：
  - 发货提醒：仅允许订单状态为“待发货/待揽收”，生成 `Uyyyymmdd...` 操作 ID，写 `shipping_urge_requests`
  - 退款申请：若存在进行中的退款（`submitted/processing`）则返回 `409` 冲突，生成 `Ryyyymmdd...` 操作 ID，写 `refund_requests`

### 3.3 `app/database.py`：数据库连接与会话生命周期

- 创建同步 `engine = create_engine(settings.database_url, pool_pre_ping=True, future=True)`
- 使用 `sessionmaker` 生成 `SessionLocal`
- `get_db()` 以 generator 形式 yield session，并在 finally 中关闭连接

### 3.4 `app/models.py`：数据模型（SQLAlchemy ORM）

- 使用 `DeclarativeBase` + typed `Mapped[]` + `mapped_column`
- 关键实体：
  - `User`（`users`）
  - `Product`（`products`，包含 `attributes_json` 的 MySQL JSON 字段）
  - `Order` / `OrderItem`（`orders` / `order_items`）
  - `LogisticsRecord` / `LogisticsTrace`（`logistics_records` / `logistics_traces`）
  - `RefundRequest`（`refund_requests`）
  - `ShippingUrgeRequest`（`shipping_urge_requests`）
- 关系：
  - `User.orders`
  - `Order.items`、`Order.logistics_records`、`Order.refund_requests`、`Order.shipping_urges`
  - `LogisticsRecord.traces`

### 3.5 `app/schemas.py`：对外 DTO（Pydantic）

- `ApiResponse`：统一响应 envelope（`code/message/data`）
- 按接口拆分 data 模型：`UserOrdersData`、`OrderDetailData`、`LogisticsData`、`ProductData` 等
- 表单类请求体：
  - `UrgeShippingRequest`（默认 `submitted_by=system`、`note` 默认文案）
  - `RefundRequestBody`（含 `reason` 必填）

## 4. 关键设计点（可复用）

- **Mock/演示定位清晰**：接口覆盖客服 demo 所需的“订单/物流/商品/操作请求”闭环，数据状态设计贴近课堂演示（待发货/运输中/已退款等）。
- **响应 envelope 一致性**：所有 API 返回 `ApiResponse`，便于前端统一处理。
- **聚合查询一次到位**：通过 `joinedload` 减少 N+1，并直接组装成前端需要的 summary/detail 结构。
- **幂等/冲突控制**：退款申请用 `409` 表示“已有进行中申请”，让上层能做更合理提示。
- **状态门禁**：发货提醒对订单状态做强约束，避免产生不符合业务的记录。

## 5. 编码风格与工程习惯

- **单模块聚合**：路由+查询+映射集中在 `app/api.py`，阅读路径短，适合 demo；规模扩大后可按 domain/service/repo 拆分。
- **命名直观**：`_get_*_or_404`、`_build_recent_*`、`create_*` 等函数名表达清晰。
- **类型注解**：`list[OrderSummaryData]`、`Mapped[list[...]]` 等在模型/业务层都有体现。
- **错误信息面向用户**：`HTTPException.detail` 直接给出中文提示，更符合教学/演示环境。

## 6. 与客服系统的对接点

- 前端（电商客服调试页）通过 Vite proxy：
  - `/commerce/users/{sender_id}/orders`、`/commerce/users/{sender_id}/products` 获取右侧“业务对象列表”
- 客服后端（智能客服）通过知识提供方/自定义 action 调用：
  - `/orders/{order_id}`、`/orders/{order_id}/logistics`、`/products/{product_id}` 等获取事实，用于 LLM 生成回复
  - `/orders/{order_id}/shipping-reminders`、`/orders/{order_id}/refund-applications` 作为任务型对话的动作落地
