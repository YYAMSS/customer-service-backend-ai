# 电商智能客服后端（customer-service-backend）技术点与架构实现要点

> 目的：为“教育版智能客服系统”复用同一套架构分层、技术选型、实现方式与编码风格提供参考基线。

## 1. 技术栈与核心依赖

- **Web 框架**：FastAPI（路由与请求/响应模型），Uvicorn（ASGI 启动）。
- **配置管理**：`pydantic-settings`，通过 `Settings(BaseSettings)` 读取项目根目录 `.env`（`SettingsConfigDict(env_file=..., extra="ignore")`）。
- **LLM 接入**：LangChain 的 `init_chat_model` 初始化 Chat Model（`model_provider="openai"`，支持 `api_key`、`base_url`、`model`，`temperature=0`）。
- **数据持久化**：SQLAlchemy Async（`AsyncSession` + `select`），以 **JSON 序列化**形式保存整份 `DialogueState`。
- **流程配置**：YAML（`FlowLoader` 加载 `flow_config/*.yml`），将“任务型对话”抽象为可配置的流程步骤与槽位定义。
- **提示词模板**：Jinja2（`.jinja2`），通过 `PromptTemplate.from_template(..., template_format="jinja2")` 加载。

## 2. 分层结构（从外到内）

### 2.1 `api` 层：HTTP 接口与 DTO

- **入口应用**：`atguigu/api/app.py`
  - 使用 FastAPI `lifespan` 管理启动/关闭。
  - 启动时完成：
    - 构建并初始化对话引擎（`build_dialogue_engine()` + `init_engine(...)`）。
    - 初始化数据库引擎（`init_db_engine()`）。
  - 关闭时释放数据库连接（`close_db_engine()`）。
- **路由**：`atguigu/api/routers.py`
  - `POST /api/chat`：将请求体转换为领域 `Message`，交给 `DialogueService` 处理，返回 `ProcessResult` 映射到 API 响应。
  - `GET /api/chat/history`：加载 `DialogueState`，从 `sessions/turns` 组装历史消息列表（包含一个 `divider` 分隔符）。
- **Schema（请求/响应模型）**：`atguigu/api/schemas.py`
  - Pydantic `BaseModel` 定义 `ChatRequest`、`ChatResponse`、`ChatHistoryResponse` 等。
  - **消息既支持文本，也支持结构化对象**（`ChatObjectPayload`：`type/id/title/attributes`）。

### 2.2 `service` 层：应用服务（有 I/O）

- `atguigu/service/dialogue_service.py`
  - 职责：对外提供“处理一条消息”的用例编排。
  - 典型流程：
    - 从仓储加载 `DialogueState`（I/O）。
    - 调用引擎处理本轮对话（纯业务逻辑）。
    - 保存 `DialogueState`（I/O）。

### 2.3 `repository` / `models` 层：状态持久化

- `atguigu/repository/dialogue_repository.py`
  - `DialogueStateRepository.load(sender_id)`：
    - 查库得到 `DialogueStateRecord`。
    - 无记录则返回新 `DialogueState(sender_id=...)`。
    - 有记录则 `json.loads(record.state_json)` 反序列化为领域状态对象。
  - `save(state)`：
    - `json.dumps(state.to_dict(), ensure_ascii=False)` 写回数据库。
    - `insert/update` 后 `commit`。
- `atguigu/models/dialogue_state.py`：SQLAlchemy 表模型（用于保存 `sender_id + state_json`）。

### 2.4 `engine` 层：无状态对话引擎（核心编排，尽量不做 I/O）

- `atguigu/engine/dialogue_engine.py`
  - 核心原则：
    - **引擎无状态**：输入 `Message + DialogueState`，输出 `ProcessResult`，并直接修改传入的 `state`。
    - **不做 I/O**：外部 API/数据库等 I/O 放在 action/provider/service 中。
  - 会话管理：
    - `DialogueConfig.session_timeout_seconds` 控制会话超时；超时/关闭会触发 `state.reset_runtime_state_for_new_session()` 并启动新会话。
  - 分轨处理：
    - `TrackType.TASK`：任务流（多轮流程/槽位收集/动作执行）。
    - `TrackType.KNOWLEDGE`：知识问答（检索/生成）。
    - `TrackType.CHITCHAT`：闲聊兜底。
  - 文本消息路径（`MessageType.TEXT`）：
    - 由 `TurnPlanner` 生成 `turn_plan`。
    - `TurnPlanValidator` 进行校验与“选轨”，失败则交给 `ClarifyResponder` 输出澄清。
    - 选中 TASK/KNOWLEDGE/CHITCHAT 后分别调用对应 handler。
  - 对象消息路径（`MessageType.OBJECT`）：
    - 将对象写入 `state.set_focused_object(...)`。
    - 依据当前是否在收集特定槽位，尝试自动生成 `SetSlotsCommand`（例如 `order_number`、`product_id`）。
    - 如果对象类型与当前上下文不匹配，会输出“需要澄清用户意图”的提示文案（例如收到订单对象但不知道要查状态/物流/退款）。

### 2.5 `planning` 层：本轮意图与执行计划（LLM 驱动）

- `atguigu/planning/planner.py`（`TurnPlanner`）
  - 面向“**本轮 Turn 计划**”的提示词驱动生成。
  - 输入给 LLM 的关键上下文包括：
    - 最近 N 轮历史（`TranscriptBuilder`）。
    - 当前激活任务（flow_id/step_id/已填槽位/当前请求槽位）。
    - 暂停任务列表（用于任务切换/恢复）。
    - 可用流程定义（从 YAML 加载后序列化为 JSON 供 LLM 参考）。
    - focused object（如果上一轮/本轮收到了对象消息）。
    - knowledge intents（可选）。
- `atguigu/planning/validator.py`（`TurnPlanValidator`）
  - 对 LLM 输出的计划做规则校验与选轨，必要时触发澄清（减少“幻觉计划/无效 flow_id/缺字段”等风险）。
- `atguigu/planning/clarify.py`（`ClarifyResponder`）
  - 将校验失败原因转为对用户的澄清提问（也是一个 LLM 生成模块）。

### 2.6 `task` 层：任务型对话（可配置 Flow + 可执行 Action）

- Flow 配置与加载：
  - `atguigu/task/flow/loader.py`：加载 `flow_config/*.yml`，并合并多个文件内的 `slots` 定义（检测重复 slot）。
  - `Flow/FlowsList/FlowSlot/FlowStep`：流程模型与步骤类型（例如 `CollectSlotStep`）。
- 执行链路：
  - `TaskHandler.handle(...)`：
    - 先 `CommandProcessor.run(...)` 执行命令（例如设置槽位、切换流程等）。
    - 再由 `FlowExecutor.run_task(...)` 跑流程推进，产出本轮消息列表。
  - Action 机制：
    - `task/actions/*`：每个 action 以 `Action.run(state, **kwargs)` 形式执行，输出 `ActionResult(slot_updates=...)`。
    - `task/actions/builtin` 与 `task/actions/custom` 分离：内置动作 + 业务定制动作。
    - 例：`LookupOrderStatusAction` 读取 `state.get_slot("order_number")`，调用外部 API，更新 `order_status`/`order_summary` 槽位。

### 2.7 `knowledge` 层：知识检索与回答（Provider + Planner + Responder）

- `KnowledgeProvider` 抽象三类来源：
  - **API**：实时结构化业务事实（如 `ProductApiProvider`、`OrderApiProvider`）。
  - **FAQ**：标准问答（当前是占位实现）。
  - **RAG**：语义检索文档（当前是占位实现）。
- Provider 特点：
  - 以 `provider_id` 注册，供 planner 选择。
  - 对外部 HTTP 调用有 `try/except` 兜底，失败时返回“可注入到 prompt 的说明性 chunk”，让 LLM 知道数据缺失而不是编造。
  - `OrderApiProvider` 使用 `asyncio.gather` 并发拉取订单与物流信息并汇总为结构化文本。

### 2.8 `chitchat` 层：闲聊兜底

- 与 `knowledge` 类似，提供 handler/responder，将不属于任务/知识的输入交给闲聊提示词生成自然回复，并可引导回业务。

### 2.9 `domain` 层：领域对象与状态机

- `Message` / `BotMessage` / `ProcessResult`：对话消息与引擎输出。
- `DialogueState`：对话状态（会话 sessions、turns、active_task、paused_tasks、slots、focused_object 等）。
- 领域层承载“状态变化”与“规则”，保证引擎可以在无 I/O 前提下驱动会话推进。

## 3. 关键设计点（可复用到教育版）

- **状态持久化的粒度**：以“整份对话状态 JSON”持久化，而不是只存消息列表。优点是容易恢复会话与任务流程；代价是需要关注 JSON 结构的兼容性。
- **无状态 Engine + 有状态 Service**：Engine 只做业务编排与状态变更；Service 负责 load/save 与事务边界，便于测试与扩展。
- **三轨并行的对话策略**：TASK（流程）、KNOWLEDGE（检索/问答）、CHITCHAT（兜底），并通过 validator 约束 LLM 选择与输出。
- **对象消息（结构化输入）加速槽位填充**：当用户从 UI 选中订单/商品对象时，可直接填槽并减少追问；若对象与当前任务不匹配，优先澄清意图。
- **可配置的任务流程**：通过 YAML 定义 flow/slot/step，新增业务流程可“以配置为主”，代码只需补 action/command/step 类型。
- **Prompt 体系化**：planning、knowledge、chitchat、clarify 分开模板，且模板路径解析做了“防目录逃逸”保护。
- **外部依赖隔离**：
  - LLM：`infrastructure/llm.py` 统一初始化注入。
  - HTTP：`infrastructure/http_client.py`（统一客户端、超时/重试可集中配置）。
  - DB：`infrastructure/database.py`（统一引擎与 session 管理）。

## 4. 编码风格与工程习惯（从代码观察到的约定）

- **类型注解充分**：`list[Type]`、`str | None`、返回类型明确；便于 IDE 与静态检查。
- **dataclass + slots**：对配置/轻量对象使用 `@dataclass(slots=True)`，减少内存与属性错误。
- **异步优先**：路由、service、engine、provider/action 多为 `async`；并发调用用 `asyncio.gather`。
- **关注职责边界**：
  - Engine 不做 I/O。
  - Repository 只关心存取与序列化。
  - Provider/Action 才做外部 API 调用。
- **失败可解释**：外部查询失败时返回“可被 LLM 使用的事实说明”，避免模型胡编数据。
- **可读性优先**：命名清晰，函数拆分到 `_build_*` / `_render_*` / `_resolve_*` 等小块。

## 5. 对教育版落地的直接映射建议（下一步会用到）

- 将 `commerce_api_base_url` 替换为 **教育业务 API**（课程/班次/订单/学习记录/退款/工单等）。
- 扩展 `object.type`：从 `order/product` 扩展到 `order/course/class/session/ticket` 等，并在对象消息路径里补充“自动填槽 + 需要澄清”的规则。
- 复用 `flow_config` 机制：把“订单查询/学习进度/退款/工单”做成 YAML flow；把具体查询/创建动作做成 custom actions。
- 保持同一套 `POST /api/chat`、`GET /api/chat/history` 接口形状与 schema（接口设计以 `doc/客服后端接口设计.md` 为准）。
