# 电商智能客服前端（customer-service-frontend）技术点与架构实现要点

> 目的：为后续在 `edu-frontend/` 复用同一套技术选型、实现方式与编码风格提供参考。

## 1. 技术栈与构建方式

- **框架**：Vue 3（Composition API，`<script setup>`）。
- **构建工具**：Vite 6（`vite` / `vite build` / `vite preview`）。
- **工程形态**：纯前端单页应用（SPA），没有路由/状态管理库，逻辑集中在一个页面组件里。
- **包管理**：仓库中同时存在 `package-lock.json` 与 `node_modules/`（说明曾用 npm 安装并提交了依赖目录；后续复用时建议避免提交 `node_modules`）。

## 2. 开发代理与后端联调方式

`vite.config.js` 配置了本地开发代理（关键点）：

- **`/api` → `http://127.0.0.1:18082`**：智能客服后端（聊天接口与 history）。
- **`/commerce` → `http://127.0.0.1:18081`**：电商业务服务（订单/商品对象列表）。
  - 同时做了 rewrite：`/commerce/...` 会去掉前缀再转发。
- **`/health` → `http://127.0.0.1:18082`**：健康检查代理。

这使得前端代码里可以直接 `fetch('/api/chat')` / `fetch('/commerce/...')`，无需写死跨域地址，也避免 CORS 问题。

## 3. 目录结构与职责划分（现状）

该前端工程非常精简，核心文件只有：

- `src/main.js`：创建并挂载 Vue App（`createApp(App).mount('#app')`）。
- `src/App.vue`：**全部 UI + 状态 + 请求 + 交互** 集中在一个组件内实现。

没有单独的：
- API client 层（如 `src/api/*`）
- 组件层（如 `src/components/*`）
- 状态管理（Pinia/Vuex）
- 路由（Vue Router）

这意味着它更像是“调试/演示前端”，而不是复杂的业务前端工程。

## 4. 核心交互模型（App.vue）

### 4.1 会话标识与消息模型

- **`senderId`**：一个可编辑输入框，决定当前会话的用户唯一标识（默认 `u1001`）。
- **消息数组 `messages`**：前端维护一份渲染用的消息列表，每条消息有：
  - `id`：`crypto.randomUUID()`
  - `role`：`user` / `bot` /（兼容）`divider`
  - `type`：`text` / `object` / `divider`
  - `text` 或 `payload`（对象卡片内容）

### 4.2 文本消息与对象消息的统一发送

- 所有发送最终走 `sendPayload(payload)`：
  - POST `chatEndpoint = '/api/chat'`
  - body 为 `{ sender_id, ...payload }`
  - 成功后把 `data.messages` 逐条 append 到聊天区
- 文本消息：
  - `sendTextMessage()` 负责读取输入框、append 本地 user 消息，然后 `sendPayload({ text })`
- 对象消息：
  - 右侧列表点击“发送订单/发送商品”，会：
    - 先 append 本地 user 的 object 消息（用于即时反馈）
    - 再发送 `sendPayload({ object: { type, id, title, attributes } })`

### 4.3 历史消息加载

- `fetchChatHistory()` GET `/api/chat/history?sender_id=...`
- 成功后调用 `setHistoryMessages(data.messages)`，将历史按 `role` 渲染回放。

> 注意：前端兼容了 `divider` 角色（用于“历史消息分隔线”），但接口文档只要求 `user/bot`。这块在教育版前端可以保留“前端自己插入 divider”的能力，但不要依赖后端返回 divider。

### 4.4 右侧“业务对象列表”面板（电商特有）

- Tab：`orders` / `products`
- 加载逻辑：`fetchSidebarData()` 并发请求：
  - `/commerce/users/{sender_id}/orders`
  - `/commerce/users/{sender_id}/products`
- 展示：每条订单/商品用卡片展示摘要，并提供按钮把对象送入会话。

该设计对应后端“对象消息”能力：把用户从 UI 选择的结构化对象直接作为消息发给智能客服，用于加速槽位填充/意图澄清。

## 5. UI/UX 设计要点（可复用）

- **整体布局**：左右两栏
  - 左：对话区（header + sender_id 控制 + message list + composer）
  - 右：业务对象列表（tabs + cards）
- **消息样式**：
  - user/bot 左右对齐、不同背景
  - 支持对象消息卡片（订单/商品 badge + 标题 + 元信息 + 金额）
- **体验细节**：
  - `watch(messages.length)` + `nextTick()` 自动滚动到底部
  - 发送态 `isSending` 禁用输入与按钮
  - 错误提示 `errorMessage` / `sidebarError`
  - senderId 变化会触发：清空会话 UI + 重新加载 sidebar + history

## 6. 编码风格（从现有代码观察）

- **Composition API + 函数式组织**：大量 `function xxx()`，命名直观（append/fetch/send/format）。
- **前端状态全部使用 `ref`**，尽量少用复杂响应式对象，降低心智负担。
- **请求使用原生 `fetch`**，无额外封装；错误处理统一 `try/catch` + `response.ok` 判断。
- **数据防御性**：对响应结构使用 `Array.isArray(data?.messages)` 等判空。

## 7. 对教育版前端的直接映射建议（用于第二步实现）

- 继续保持“**聊天区 + 右侧业务对象列表**”两栏布局：
  - 电商的订单/商品 → 教育的 **订单/课程/班次**（以及可选：学习记录/工单）
- Vite proxy 将：
  - `/api` 指向教育智能客服后端（你当前 `edu-service-backend` 默认 `8010`）
  - `/edu`（或其它前缀）指向教育业务数据服务（若有）
- 仍然使用 “文本消息 + 结构化对象消息” 两种输入方式，复用后端的 object schema（`type/id/title/attributes`）。
