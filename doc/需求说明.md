# 教育智能客服系统项目实战需求说明
## 1. 项目背景与目标
### 1.1 项目背景
在线教育平台面向机构、教师、学员和运营团队提供教学与经营支撑能力，覆盖课程供给、招生转化、交易支付、学习过程、互动服务和售后处理等完整业务链路。随着业务规模扩大，学员在课程咨询、订单查询、学习跟踪、售后处理等方面的服务请求量持续增长，需要一套智能客服系统来承载这些高频服务场景。

### 1.2 项目目标
构建一个面向在线教育行业的智能客服系统，支持以下核心目标：
- 自动识别学员的咨询意图，提供课程咨询、订单查询、学习进度查询等问答能力
- 支持退款申请、工单提交等需要多轮引导的任务型对话
- 在多轮对话中持续收集和利用上下文信息（学员身份、订单号、班次等）
- 所有对话状态可持久化，支持会话恢复和历史回溯

## 2. 用户角色
### 2.1 学员
核心服务对象。通过智能客服咨询课程信息、查询订单状态、了解学习进度、发起退款或投诉等。

### 2.2 客服人员
承接系统无法自动处理的问题，负责人工兜底、复杂售后处理和工单跟进。

### 2.3 运营人员
维护知识库内容、FAQ 条目和服务流程配置，保障客服系统稳定运行。

## 3. 功能需求
### 3.1 信息检索
系统应能从业务数据和知识库中检索相关信息并以自然语言组织回复，支撑课程咨询、订单查询、学习进度查询、政策咨询等场景。
- 实时业务数据检索：查询课程、班次、订单、学习记录等动态数据
- FAQ 检索：匹配退款政策、开课政策等相对固定的制度性问题
- 知识库检索：检索平台规则、使用指南等通用内容

### 3.2 任务型对话
系统应能通过多轮对话引导用户完成需要收集多项信息才能处理的业务任务。
- 支持订单状态查询：收集订单号 → 查询并返回订单状态与报名信息
- 支持学习进度查询：收集班次信息 → 查询并返回考勤、视频完成情况、作业提交情况、考试成绩等学习数据
- 支持退款申请：收集订单号、退款原因、退款类型 → 创建退款申请并返回处理结果
- 支持工单提交：收集工单类型（售后/投诉/退款）、关联订单号、问题描述 → 创建工单并返回工单编号与处理预期

### 3.3 对话管控
系统应具备完整的多轮对话管理能力。
- 支持槽位收集：在任务流程中主动向用户追问缺失的信息（如订单号、班次名称、退款原因）
- 支持上下文保持：在多轮对话中持续利用已收集的学员信息、订单信息、班次信息
- 支持流程切换：当前任务执行中，用户发起新任务时，暂存当前任务并切换到新任务
- 支持流程恢复：新任务完成后，恢复之前暂存的任务继续处理
- 支持流程取消：用户主动取消当前任务时，清理对应状态并确认取消

### 3.4 闲聊兜底
当用户输入不属于明确的业务任务或信息查询时，系统应能给出自然、友好的闲聊回复，并在适当时候引导用户回到业务场景。

## 4. 典型业务场景
### 4.1 课程咨询
学员询问课程内容、适用人群、班次安排、价格等信息时，系统应结合课程与班次数据返回清晰回复。
```
学员：Python 全栈课程大概是什么情况？
系统：返回课程概述、适用人群、在售班次及价格，并引导学员选择具体班次了解更多
```

### 4.2 订单查询
学员提供订单号后，系统应返回订单状态、支付情况、报名课程等信息。
```
学员：帮我查下订单 ORD20240401005
系统：返回订单状态（待支付/已支付/已取消等）、报名课程名称、金额、支付时间
```

### 4.3 学习进度查询
学员询问某班次的学习情况时，系统应返回考勤、视频、作业、考试等多维度进度数据。
```
学员：我在 Python 全栈第5期的学习进度怎么样？
系统：返回出勤次数/缺勤次数、视频完成数/总数、作业提交数/总数、考试参加数/总数
```

### 4.4 退款申请
学员发起退款时，系统应收集订单号和退款原因，创建退款申请并反馈处理结果。
```
学员：我要退款
系统：收集订单号 → 收集退款原因 → 确认退款类型 → 提交退款申请 → 返回确认信息
```

### 4.5 工单提交
学员遇到售后问题、投诉或需要人工介入时，系统应收集必要信息并创建工单。
```
学员：第三章的视频一直加载不出来，我要投诉
系统：确认工单类型 → 收集关联订单号 → 收集问题描述 → 创建工单 → 返回工单编号
```

## 5. 对话交互要求
### 5.1 消息处理
- 支持接收文本消息和业务对象消息（如学员发送一个班次或订单对象）
- 收到业务对象消息时，应自动提取对象信息并尝试匹配当前任务所需的槽位

### 5.2 响应模式
- 支持非流式响应（一次性返回完整回复）
- 支持流式响应（通过 SSE 逐字返回回复内容）

### 5.3 会话管理
- 支持创建新会话
- 支持获取会话历史消息
- 支持获取当前会话状态（当前激活的任务流程、已收集的槽位信息）
- 会话状态应持久化存储，支持会话恢复

### 5.4 意图澄清
当系统无法准确判断用户意图时，应主动向用户澄清，而不是直接给出错误回复。澄清场景包括：
- 用户输入同时匹配多个意图
- 用户输入过于模糊，无法确定具体需求
- 当前任务需要的关键信息缺失
- 用户输入与当前任务上下文存在冲突

## 6. 非功能需求
### 6.1 可靠性
- 单次对话处理失败不应影响整体服务可用性
- 对外部业务 API 的调用应具备超时和异常处理能力
- 对话状态保存失败时应有降级处理

### 6.2 可观测性
- 应能追踪每次对话的处理轨迹（意图识别结果、进入的轨道、执行的动作、生成的回复）
- 应能记录关键节点的执行状态和耗时
- 工单处理过程应支持日志记录与审计

### 6.3 准确性
- 查询结果应基于实时业务数据，避免返回过时或错误信息
- 任务流程中的槽位收集应准确提取用户输入中的关键信息
- FAQ 和知识库检索应优先返回最匹配的内容

### 6.4 可扩展性
- 新增任务流程应可通过配置方式实现，无需修改核心代码
- 新增知识意图类型应可灵活扩展
- 新增业务 API 调用应可方便集成

## 7. 项目交付物
- 对话接口与会话管理能力
- 一套可运行的业务样本数据与配置
- 前端调试/演示页面
- 项目说明文档与启动说明

## 8. 验收标准
1. 能够完成课程咨询、订单查询、学习进度查询等基础问答
2. 能够在多轮对话中持续收集和使用学员身份、订单号、班次等槽位数据
3. 能够完成退款申请、工单提交的完整多轮任务流程
4. 能够完成一次完整的"用户提问 → 意图识别 → 业务查询/流程处理 → 生成回复 → 状态更新"闭环演示
5. 对话状态可持久化，支持会话恢复
